## Decision tree

1. **Vercel + Supabase integration:** keep the default launch flow.
2. **Non-Vercel + Supabase:** use infra targets with Supabase mode.
3. **Non-Vercel + Postgres+S3:** use infra targets with S3 mode and an external scheduler.

---

## Storage modes

| Mode | Required variables | Storage backend | Scheduler |
| --- | --- | --- | --- |
| **Supabase mode** | `POSTGRES_URL`, `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY` | Supabase Storage (`kuest-assets`) | Built-in (`pg_cron` via `npm run db:push`) |
| **Postgres + S3 mode** | `POSTGRES_URL`, `S3_BUCKET`, `S3_ACCESS_KEY_ID`, `S3_SECRET_ACCESS_KEY` (+ optional `S3_*`) | S3-compatible object storage | External/platform scheduler required |

---

## Operating principles

- Always start from base secrets in [Environment Variables](/docs/owners/environment-variables).
- Use immutable image references in production (`@sha256:digest` preferred).
- Set `SITE_URL` to your canonical public URL if you are deploying outside Vercel.
- Choose one scheduler backend only (avoid duplicate schedulers).
